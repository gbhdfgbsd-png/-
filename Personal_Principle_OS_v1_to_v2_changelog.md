# Personal Principle OS: v1 → v2 Changelog

## 变更驱动：七维度系统评估

本次重大版本升级基于对v1的七维度深度评估，每个变更点都有明确的问题归因和设计理由。

## 详细变更说明

### 变更1：确认机制翻转

**v1问题：** 系统每个输出都需要用户`[✓确认] [✏️修正]`才生效，导致9类确认行为持续占用认知资源。

**v2设计：**
- 所有系统输出**默认生效**
- 每类确认行为有独立的**毕业条件**（基于用户纠正率自动判断）
- 毕业前：需确认（校准期）
- 毕业后：默认生效，用户只在不同意时介入

**毕业条件示例：**

| 确认行为 | 毕业条件 |
|---------|----------|
| 决策记录结构化 | 连续20条未修正 |
| 维度分类 | 某Domain内连续10条未改 |
| 试金石格式化 | 连续15条未调 |
| 模式识别 | 前3个confirmed模式均认可 |
| TFRF评分 | 手动调分<2次/月 |

### 变更2：三态系统运行模式

**v1问题：** 单一运行模式，无法适应用户状态波动。

**v2设计：**

```yaml
system_states:
  calibration_mode:  # 前30-60天
    description: "用户教系统，每类确认有退出跑道"
    user_effort: "每天5-10分钟"
    exit_condition: "所有确认行为毕业"
    
  cruise_mode:  # 校准完成后的常态
    user_daily: "随时输入(0摩擦) + 固定时间看Briefing(<2分钟)"
    user_monthly: "Sync Ritual(15分钟)"
    system_auto: "解析/分类/关联/检测/压缩"
    
  survival_mode:  # 自动检测触发
    trigger: "7天无输入 或 用户手动设置"
    behavior: "只存储输入，不做任何处理/推送/确认请求"
    recovery: "恢复输入后自动回升，积压批量处理"
```

### 变更3：每日Briefing设计

**v1问题：** 多个独立推送触点（张力报告、回填提醒、模式确认等）分散打扰。

**v2设计：** 合并为单一Briefing，按重要度排序：

| 优先级 | 内容类型 | 出现条件 |
|--------|---------|---------|
| 1 | 系统自我修改通知 | 系统改了自己的规则时 |
| 2 | 需要你输入的卡点 | 解析confidence<0.5的记录 |
| 3 | 发现/趋势摘要 | 有新emerging模式或显著趋势时 |
| 4 | 即将到期的回填 | 回填点前1天 |
| 5 | 系统健康异常 | 指标超阈值时 |

**约束：**
- 总条数≤5
- 每条一句话+可展开详情
- 总阅读时间<2分钟
- 无重要信息的日子=不推送

### 变更5：双通道模式检测

**v1问题：** 模式引擎仅从stated_reason提取模式，可能被事后合理化污染。

**v2设计：**

```yaml
pattern_detection_dual_channel:
  channel_1_reason_based:
    source: "stated_reason字段"
    method: "语义聚类，找重复逻辑"
    risk: "可能被事后合理化污染"
    
  channel_2_behavior_based:
    source: "选择本身的结构特征"
    method: "不看理由，只看选项客观属性（reversibility/stakes/time_horizon/effort_required）"
    
  cross_validation:
    both_channels_agree: "高置信度模式"
    only_channel_1: "⚠️ 可能是合理化产物，降权处理"
    only_channel_2: "🔥 用户无意识的真实倾向——最高价值发现"
```

### 变更6：决策前直觉捕获

**v1问题：** 所有数据都是决策后录入，无法区分"真实驱动力"和"事后解释"。

**v2设计：** 仅在L5咨询（重大决策）时触发：

```yaml
pre_analysis_capture:
  trigger: "用户发起L5决策咨询时"
  question: "在你看系统分析之前——你现在倾向哪个选项？信心多大(1-10)？"
  purpose: 
    - 建立反事实基准（没有系统你会怎么选）
    - 捕获直觉vs理性的真实张力
    - 为系统价值追踪提供数据
```

### 变更7：回填机制分级

**v1问题：** 所有决策统一回填时间点，高stakes决策过早判断被短期情绪污染。

**v2设计：**

```yaml
backfill_timing:
  stakes_trivial: no_backfill
  stakes_low: 7_days
  stakes_medium: 30_days
  stakes_high: [30_days_feeling_only, 90_days_assessment]
  stakes_critical: [30_days_feeling, 90_days_preliminary, 180_days_assessment, 365_days_final]

backfill_rules:
  high_and_critical_first_backfill:
    allowed_responses: ["记录当前感受", "记录客观变化"]
    NOT_allowed: ["正向/负向判断"]  # 太早判断会被情绪污染
  
  every_positive_backfill:
    forced_question: "这个决策有没有任何小的负面后果，即使总体正向？"
```

### 变更10：复杂度硬上限

**v1问题：** 软建议"active原则>50条→触发压缩建议"，用户可以无限忽略。

**v2设计：**

```yaml
complexity_budget:
  active_principles:
    hard_cap: 40
    at_cap: "新原则入库→必须同时提案一条旧原则退役/合并/降级"
  active_hypotheses:
    hard_cap: 100
    at_cap: "TFRF最低10条自动降级为archived"
  active_tensions:
    hard_cap: 10
    at_cap: "最旧未解决张力自动降级为observation"
  active_axes:
    hard_cap: 25
    at_cap: "触发合并扫描"
```

### 变更11：决策日志压缩周期

**v1问题：** 决策日志只增不减，一年后700+条且大部分永远不会被访问。

**v2设计：**

```yaml
log_compression:
  cycle: "每90天"
  process:
    1: "识别已被模式引擎消化且产出confirmed原则的决策"
    2: "这些决策→压缩为一行摘要（日期+选择+结果）"
    3: "原始详细记录→删除"
    4: "保留：未消化的 + 有回填标记的 + high/critical stakes的"
```

### 变更12：关联剪枝

**v2新增：**

```yaml
association_pruning:
  scan_cycle: "每180天"
  dormant_condition: "该关联180天内未被任何层引用 + 两端节点无活跃张力"
  dormant_to_delete: "再过180天仍为dormant → 删除"
```

### 变更14：主动去权威化机制

**v1问题：** 只有被动防护（Override Protocol），无法抵抗用户对系统的渐进依赖。

**v2新增三个主动机制：**

**A. 季度自我暴错：**
```yaml
quarterly_self_challenge:
  task: "找出系统分析倾向A但用户选B且结果正向的案例"
  presentation: "本季度你有X次违背系统方向且结果更好。系统在这些场景下不如你的直觉：[...]"
```

**B. 随机静默期：**
```yaml
silence_period:
  frequency: "每2-3个月一次"
  duration: "5-7天"
  behavior: "L5不可用，无Briefing推送，用户纯靠直觉"
  post_silence: "对比静默期决策质量与有系统时的质量"
```

**C. 张力框架轮换：**
```yaml
tension_framing:
  rotate_randomly:
    frame_A: "PP-003偏离率40%。你的行为在偏离原则。"
    frame_B: "你的行为呈现新模式。PP-003可能不再描述真实的你。"
    frame_C: "数据不一致。可解读为'你在偏离'或'原则在过时'。你怎么看？"
```

### 变更15：系统权威监控器

**v2新增：**

```yaml
system_authority_monitor:
  metric: "过去30天用户选择系统主要论证方向的比例"
  healthy_range: "60%-80%"
  alert_threshold: ">85%持续60天"
  alert_action: "建议下次先自己想清楚方向再看系统分析"
```

### 变更20：脉冲式运行节奏

**v1问题：** 固定频率运行（每天/每周/每月），不匹配人生的决策密度波动。

**v2设计：**

```yaml
rhythm_detection:
  low_power_trigger: "连续14天输入<2条"
  full_power_trigger: "7天内输入≥5条 或 发起L5咨询 或 stakes=high"
  
  low_power_mode:
    - 不推送Briefing
    - 模式引擎暂停
    - 只做：存储+长周期回填提醒
    
  full_power_mode:
    - 全层激活
    - Briefing恢复
    - 主动关联推送
    
  transition_auto: true  # 不需要用户手动切换
```

### 变更22：重激活校验协议

**v2新增：** 系统从低功耗切换到全功率时的第一个动作。

```yaml
reactivation_protocol:
  trigger: "低功耗→全功率切换"
  action:
    - 取top 5最可能被调用的原则
    - 对用户提问："距上次活跃X个月。这3条核心原则的前提还成立吗？"
    - 用户回答"是/部分变了/完全不同了"
  response:
    "是": "正常激活"
    "部分变了": "变化原则降权，密集期后重新审视"
    "完全不同了": "所有原则标记待重新验证，进入重校准模式"
```

### 变更23：Kill Switch机制

**v2新增：** 系统可证伪性的终极保障。

```yaml
system_value_falsification:
  core_claim: "本系统帮助用户做出更好的决策"
  
  kill_switch_conditions:  # 任意2条满足→系统主动建议关闭
    - "连续12个月L5咨询后改变选择且结果正向的次数=0"
    - "连续12个月张力报告导致的原则修订或行为调整=0"
    - "静默期对照显示决策质量无差异，连续2次"
    - "用户月度输入量连续6个月<3条"
  
  action: "系统主动建议：关闭/大幅简化/降低投入"
```

### 变更24：各层可证伪声明

**v2新增：**

| 系统层 | 可证伪声明 | 无用判定 |
|--------|-----------|---------|
| L2模式引擎 | 发现用户盲区 | 用户对呈现的模式反应全是"我知道"→12个月无盲区发现 |
| L4张力检测 | 促使行为/原则修正 | 6个月内零次修正触发 |
| L5决策辅助 | 改善决策质量 | 反事实追踪显示净正贡献=0 |
| L6演化追踪 | 帮助理解变化方向 | 报告未导致任何认知或行动改变 |
| L1试金石库 | 假说被有效调用 | 年度被L5调用且影响决策的假说<5条 |

## 设计哲学层面的变更

| 维度 | v1隐含哲学 | v2显式哲学 |
|------|-----------|-----------|
| 用户角色 | 系统的管理者 | 系统的使用者（只负责输入+提问） |
| 系统角色 | 需要确认的助手 | 默认自治+定期汇报的自运行系统 |
| 复杂度观 | 越完备越好 | 有硬上限，遗忘和增长同等重要 |
| 时间观 | 匀速运行 | 脉冲式，随生活节奏呼吸 |
| 价值观 | 隐含"永续使用=成功" | 显式"让你不再需要系统=终极成功" |
| 诚实观 | 信任用户输入 | 交叉验证，在偏差中寻找真实信号 |
| 系统自信 | 不敢建议用户放弃 | 有Kill Switch，敢承认自己可能没用 |
